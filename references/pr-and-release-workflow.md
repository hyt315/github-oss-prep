# PR、验证与发布门禁

## AI 改代码是不是主流

AI 直接修改代码已经常见；成熟协作的关键不是“是否由 AI 写”，而是变更是否经过可审查的分支、自动检查和人工合并。默认路径是：

`AI 研究与计划 → 工作分支 → 修改 → 测试 → Draft PR → 人工审查 → CI → 合并 → Tag/Release`

## 两种模式

| 模式 | 适用 | 默认动作 |
|---|---|---|
| `public-safe` | 已公开、多人协作、准备长期维护 | 分支 + Draft PR + CI + 人工合并 |
| `solo-fast` | 单人引导期、文档小修、可快速回滚 | 展示 diff 和测试后，可经明确同意直推 |

以下改动即使在单人项目也优先走 PR：认证/秘密、许可证、构建与发布、依赖升级、可执行脚本、数据迁移、权限设置。

## 发布前门禁

### P0：任一失败即停止

- 无秘密、私人数据或真实凭据进入当前树与待推送历史；
- 所有代码和资产有分发权，第三方许可证与 NOTICE 完整；
- 干净 clone 可按 README 安装并运行最小示例；
- 测试、构建、lint 和项目自带验证通过；
- 仓库可见性、Owner、名称、许可证已经用户确认。

### P1：公开前应完成

- README 首屏能说明用户、问题、结果与安装方式；
- 支持平台、依赖版本、已知限制、排障与卸载方式明确；
- Issue/PR 模板、CONTRIBUTING、SECURITY 和反馈渠道可用；
- CI 在默认分支与 PR 上运行，Ruleset 要求 PR 和必要检查；
- 版本号、CHANGELOG、Badge、Tag、Release 标题与安装链接一致。

## AI 变更的审查摘要

每次请求合并或直推前输出：

1. 改了什么及为什么；
2. 文件级 diff 摘要；
3. 实际运行的命令与结果；
4. 未运行的验证及原因；
5. 风险、回滚方式和是否涉及外部状态；
6. 需要用户确认的精确动作。

## 自修改保护

当本 Skill 准备、修改或发布自身时：

- 在独立 clone/worktree 中操作，不直接覆盖已安装且正在运行的副本；
- 先验证新副本，再更新安装副本；
- 不从正在处理的项目中读取或保存认证凭据；
- 源码推送、Tag、Release 分开验证，任何一步失败都保留可恢复状态。
